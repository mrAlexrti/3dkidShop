# 3D Kid — журнал редизайна (WORK-LOG)

Ветка: `redesign-3dkid-style`. Цель: современный playful/pastel/kawaii/3D-редизайн на основе
`ескиз.jpg`, **без слома бизнес-логики** (LiqPay, Nova Poshta, Telegram, Resend, GitHub
Actions, checkout, статусы заказов, API routes, env, URL). Документ живой — обновляется после
каждого этапа.

Связанная спека: [2026-07-08-redesign-3dkid-design.md](./2026-07-08-redesign-3dkid-design.md).

## Этап 1 — Фундамент ✅

### Сделано
- **Серверный бэкенд i18n.** Источник языка — cookie `locale` (по умолчанию `ua`). Новый
  `src/lib/i18n-server.ts` (`getLocale`/`getT`) для server-компонентов и админки. Переключатель
  языка пишет cookie + `router.refresh()`. Middleware не трогали.
- **Убран хардкод storefront.** Компоненты `home/*` и `catalog/*`, хардкодившие **русский**
  текст, подключены к словарю (UA/EN).
- **Шрифты.** `src/lib/fonts.ts` — 5 кириллических тем; активная читается из
  `SiteContent['site_font']` (`src/lib/site-settings.ts`), подключается динамически через
  `<link>` Google Fonts в корневом layout, применяется через CSS-переменные
  `--font-body`/`--font-display`.
- **Категории.** Prisma: добавлено **только** `Category.parentId` (self-relation,
  `onDelete: SetNull`). Добавлен `updateCategory` (его не было), `create/updateCategory`
  принимают `parentId` + `imageUrl`. Seed идемпотентно засеивает канонический набор.
- **Бренд** STIKR → 3D Kid в сайдбаре админки.

## ❗ Что НЕ сделано по ТЗ и почему

Здесь — все отклонения от буквы ТЗ с причинами. Блокеров нет; это осознанные решения либо
пункты, требующие вашего действия/разрешения.

1. **«Прибрати дублікати категорій» — не сделано автоматически.**
   **Причина:** это деструктивная операция на живых данных прода, а удаление категории с
   привязанными товарами упадёт по внешнему ключу (`Product.categoryId` обязателен, без
   каскада). У меня нет доступа к проду, и авто-удаление живых данных недопустимо.
   **Что сделано вместо:** канонический набор (идемпотентный seed) + навигация content-driven +
   в админке появились edit/delete/подкатегории для ручного дедупа. **Дополнительно предоставлен
   безопасный скрипт** `scripts/dedupe-categories.ts` (перепривязка товаров в канонические
   категории → удаление старых; dry-run по умолчанию, `--apply` для применения) — см.
   [dedupe-categories.md](./dedupe-categories.md). Запускается вами осознанно на проде (после бэкапа).

2. **Локализация — покрытие ~90%, есть хвост.** Не локализованы: `admin/orders/[id]` (детали
   заказа), `login-form`, `create-ttn-button`, `order-status-select`.
   **Причина:** это операционные/auth-экраны, плотно завязанные на **логику статусов заказов**,
   которую ТЗ прямо запрещает менять; чтобы не задеть статусную логику и не раздувать объём,
   отложил их. Делается тем же паттерном `getT`/`useLangStore` — по запросу докручу.

3. **`getOrderStatusLabel` не тронут.**
   **Причина:** это часть логики статусов заказов (ТЗ: «не змінювати логіку статусів
   замовлення»). Ярлыки статусов оставлены как есть, чтобы не рисковать статусной логикой.

4. **Реальная загрузка файлов картинок не делалась (только URL).**
   **Причина:** ваше согласованное решение. Инфраструктуры загрузки в проекте нет, а real
   upload (Supabase Storage) требует новых env-переменных (`SUPABASE_URL`, service key), что
   запрещено ТЗ («не змінювати env variables»). Сделано URL + preview + fallback.

5. **Шрифт хранится в `SiteContent`, а не в новой модели `SiteSettings`.**
   **Причина:** ваше согласованное решение. `SiteContent` уже хранит настройки сайта;
   отдельная модель = дублирование сущности и лишняя миграция. Функционально «глобальный выбор
   шрифта через настройки» выполнен.

6. **Редактирование товара не меняет его картинку.**
   **Причина:** пре-существующее ограничение (`updateProduct` не пишет `images`), вне скоупа
   редизайна — не трогал, чтобы не менять поведение. Требуется — отдельная задача.

7. **Главная потеряла ISR (стала dynamic).**
   **Причина:** побочный эффект серверной i18n — корневой layout читает cookie (`cookies()`),
   что переводит все страницы в dynamic. Это осознанный компромисс ради локализации
   server-компонентов/админки. Функционал не ломается, теряется только CDN-кэш главной. При
   необходимости ISR главной есть альтернатива (держать её локаль только клиентской).

8. **Поэтапные «стопы» после этапов 1–2, этапы 3–4 сделаны пакетом.**
   **Причина:** по вашему указанию «сделай все правки, потом посмотрим».

## ⚠️ Особенности и важные нюансы разработки (must-know)

1. **`npm install` только с `--legacy-peer-deps`.** React 19 RC конфликтует по peer-зависимостям
   с `framer-motion`. Обычный `npm install` падает. Касается и CI/Vercel при чистой установке.
2. **Локальный `next build` требует реальной `DATABASE_URL`.** `sitemap.ts` и
   `generateStaticParams` в `/product/[slug]` читают БД на этапе build (пре-существующее).
   Верификация без БД: `npm run lint` ✅, `npx tsc --noEmit` ✅, а `next build` доходит до
   `✓ Compiled successfully`. С реальной БД (или локальной dev-БД) build проходит полностью.
3. **Серверная i18n делает страницы dynamic.** См. пункт 7 выше.
4. **Гидратация EN.** zustand-persist на сервере не запускается → серверный рендер клиентских
   компонентов всегда `ua`; на клиенте persist может дать `en` → короткий флеш для
   EN-пользователей (маскируется `suppressHydrationWarning`, согласуется через cookie +
   `router.refresh()`). Для дефолтных `ua`-пользователей флеша нет.
5. **Названия категорий НЕ локализуются.** В схеме одно поле `name` — названия категорий всегда
   из БД (по-украински). i18n покрывает только UI-chrome, не контент (осознанный выбор, чтобы
   не раздувать схему переводами контента).
6. **Навигация content-driven → дубликаты убираются вручную.** Header/главная показывают
   реальные категории верхнего уровня из БД. На проде это ваши текущие категории; канонический
   набор (4 категории + 3 подкатегории) формируется через админку. Seed добавляет канонические
   поверх демо только в dev.
7. **Картинки категорий — обычный `<img>`, не `next/image`.** URL категории может быть с любого
   хоста, а `next/image` падает на не-allowlisted домене (`next.config` разрешает только
   unsplash/supabase). Поэтому в `CategoryForm` (preview), таблице категорий и карточках
   главной — `<img>` + fallback через `onError` (+ `eslint-disable no-img-element`). Картинки
   **товаров** оставлены на `next/image` (пре-существующее, хосты ожидаемо allowlisted).
8. **Выбор шрифта — runtime через `<link>`.** `next/font` статичен на build и не умеет
   runtime-выбор из БД без загрузки всех семейств; поэтому грузится только выбранная тема через
   `<link>` в layout. React 19 хойстит `<link>` в `<head>` и дедуплицирует по href.
9. **Высота Header зашита в padding.** `(shop)/layout.tsx` держит `pt-[150px] md:pt-[228px]`,
   что должно совпадать с `min-h` Hero + высотой полосы категорий в `header.tsx`. Меняешь
   высоту Hero/nav — синхронно правишь padding (значения могут потребовать ±10px после
   визуального QA).
10. **`prisma generate` после изменения схемы** — иначе TS не видит новые поля
    (`parentId`/`children`).
11. **`db:seed` — только dev.** Seed содержит демо-товары и **неидемпотентные** отзывы
    (`review.create` дублирует на каждом прогоне). На проде не запускать.
12. **Редактирование товара не меняет картинку** (`updateProduct` не пишет `images`) —
    пре-существующее ограничение, не трогал.
13. **Селектор шрифта** сохраняется через общий `updateSiteContent` (upsert любых ключей).
    После сохранения `revalidatePath("/")` → layout перечитывает `site_font`.
14. **Локальное превью.** Для визуальной проверки поднята изолированная dev-БД `threedkid_dev`
    на локальном Postgres (`postgres/postgres`), `db push` + `db seed`, `npm run dev`. `.env`
    для превью — **gitignored**, не для коммита. `tsx` не читает `.env` → seed запускать с
    явным `DATABASE_URL=... npx tsx prisma/seed.ts`. Админка за auth (login требует `ADMIN_*` env).
15. **Удалён мёртвый код** `src/components/home/hero.tsx` (нигде не рендерился, содержал
    хардкод RU и старый бренд STIKR).
