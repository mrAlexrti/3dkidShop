# Дедуп категорий — `scripts/dedupe-categories.ts`

Безопасный скрипт для устранения дубликатов/старых категорий: переносит товары из старых
категорий в канонические, затем удаляет опустевшие старые. Закрывает пункт ТЗ «прибрати
дублікати категорій» контролируемым способом, без риска потерять товары.

> **Почему это отдельный скрипт, а не авто-миграция.** Удаление категории — деструктивная
> операция на живых данных, а у категории с товарами есть внешние ключи (`Product.categoryId`).
> Автоматически удалять продовые данные нельзя. Скрипт запускается **вами, осознанно**, сначала
> в режиме предпросмотра.

## ⚠️ Перед запуском на проде
Сделайте бэкап БД (например, дамп через провайдера/`pg_dump`). Отката «одной кнопкой» у скрипта
нет — он рассчитан на предварительный dry-run + бэкап.

## Что делает
1. Гарантирует существование канонического набора (идемпотентный upsert):
   `3d-toys`, `other-3d` (+ подкатегории `buttons`, `crocs-jibbitz`, `earphone-accessories`),
   `shoppers`, `courses-3d`.
2. Для каждой пары `from → into` из `MERGE_MAP`: переносит **все товары** из `from` в `into`
   (в транзакции), затем удаляет `from`.
3. Печатает отчёт: план слияний, пропуски и **незамапленные** неканонические категории.

## Гарантии безопасности
- **DRY-RUN по умолчанию** — без флага ничего не меняется, только печатается план.
- Применение только с `--apply` (или env `CONFIRM=1`).
- Никогда не удаляет **каноническую** категорию (`from` не может быть каноническим).
- `into` обязано быть **каноническим** — иначе пара пропускается.
- Категорию с **подкатегориями** не удаляет (чтобы не осиротить детей) — пропускает с предупреждением.
- **Незамапленные** неканонические категории не трогает — только показывает в отчёте.
- **Идемпотентность**: повторный запуск после применения безопасен (старых slug уже нет → «нет к слиянию»).
- Товары не удаляются, а перепривязываются → заказы (`OrderItem`) не затрагиваются.

## Как пользоваться

### 1. Посмотреть план (безопасно)
```bash
# если есть .env с DATABASE_URL:
npx tsx scripts/dedupe-categories.ts
# или npm-скрипт:
npm run db:dedupe-categories
# либо передать URL явно (tsx не читает .env сам, но скрипт умеет подхватить .env):
DATABASE_URL="postgresql://..." npx tsx scripts/dedupe-categories.ts
```
Отчёт покажет реальные категории вашей БД, включая **незамапленные** — по ним вы поймёте, что
ещё есть на проде.

### 2. Настроить соответствия
Откройте `scripts/dedupe-categories.ts`, отредактируйте `MERGE_MAP` под свои реальные категории:
```ts
const MERGE_MAP: { from: string; into: string }[] = [
  { from: "stickers", into: "3d-toys" },   // товары из «stickers» → «3D іграшки»
  { from: "posters",  into: "other-3d" },  // и т.д.
  // добавьте свои пары из отчёта dry-run
];
```
`from` — slug старой/дублирующей категории; `into` — slug канонической, куда перенести товары.

### 3. Применить
```bash
npx tsx scripts/dedupe-categories.ts --apply
# или: DATABASE_URL="postgresql://..." npx tsx scripts/dedupe-categories.ts --apply
```

### 4. После
Навигация/каталог/главная content-driven — сразу покажут только оставшиеся (канонические)
категории. Если нужен redeploy для сброса кэша страниц — сделайте `redeploy` (данные уже
изменены в БД).

## Что скрипт НЕ делает
- Не удаляет незамапленные и канонические категории.
- Не сливает категории с подкатегориями (сначала разберитесь с детьми вручную/через `MERGE_MAP`).
- Не трогает товары, заказы, изображения (только меняет `Product.categoryId`).

## Проверено
Прогнан на изолированной dev-БД: dry-run → корректный план; `--apply` → 12 демо-товаров
перенесены, 4 старые категории удалены; повторный dry-run → «нет к слиянию» (идемпотентно).
